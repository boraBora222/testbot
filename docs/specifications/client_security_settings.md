# ТЗ: Безопасность и настройки клиента (пакет 2)

## 1. Цель

Реализовать единый продуктовый блок в профиле клиента для B2B-сервиса криптообмена:

1. отображение лимитов и их использования;
2. базовые настройки пользовательских уведомлений;
3. whitelist адресов кошельков и использование whitelist при создании заявки.

Этот пакет соответствует исходному составу второго блока и закрывает уже существующие пробелы системы: на фронте whitelist и уведомления сейчас существуют как заглушки, в API нет нужных полей, в модели пользователя нет настроек, а при создании заявки адрес не проверяется по whitelist.  

---

## 2. Границы пакета

### Входит в пакет

* прогресс лимитов в профиле и в Telegram-боте;
* сохранение и изменение пользовательских настроек уведомлений;
* CRUD whitelist-адресов клиента;
* ручная модерация whitelist-адресов менеджером;
* выбор адреса из whitelist при создании заявки;
* запрет создания заявки на адрес, отсутствующий в активном whitelist;
* аудит изменений лимитов на стороне админки.  

### Не входит в пакет

* SMS-канал;
* клиентская история изменений лимитов;
* SLA-аналитика, эскалации, приоритизация очереди модерации;
* CRM/ticket-интеграции;
* перевод сущностей с уровня пользователя на уровень компании;
* сложные сценарии массового управления whitelist.

Эти пункты были добавлены или расширены в исходной редакции ТЗ, но не являются обязательной частью второго пакета “безопасность и настройки клиента”.  

---

## 3. Зафиксированные бизнес-правила

### 3.1. Scope сущностей

В версии v1 все лимиты, настройки уведомлений и whitelist привязаны к `user_id`, а не к компании. Это соответствует текущей архитектуре, где профиль, бот и админские операции уже завязаны на пользователя. Расширение до company-scoped модели не входит в этот пакет.  

### 3.2. Правило по whitelist

Сделка может быть создана только на адрес из активного whitelist пользователя.
Ручной ввод адреса без активного совпадения допустим только как способ выбора адреса для последующей проверки, но не как способ обойти whitelist. Если адрес отсутствует в whitelist, пользователь должен сначала добавить его в whitelist и дождаться модерации.

Это правило фиксируется как единое для всех клиентских интерфейсов и заменяет противоречивые формулировки вида “проверка после заявки” и “только для первой сделки”. Публичные тексты в FAQ/UI должны быть приведены к одной формулировке. Базой для этого правила является уже существующий FAQ: “Deals can only be made to addresses from this list.”  

### 3.3. Лимиты

Система хранит и показывает дневной и месячный лимит, а также фактическое использование по каждому из них.
Зафиксированные дефолты по дневным лимитам:

* `basic` — 1 000 000 ₽ в день;
* `extended` — 10 000 000 ₽ в день;
* `corporate` — задаётся вручную.

Месячный лимит хранится в системе и настраивается администратором. Жёсткие дефолтные месячные значения в этом ТЗ не фиксируются, чтобы не вводить противоречие с уже опубликованными правилами, где определены только дневные лимиты.  

### 3.4. Поведение при превышении лимита

В рамках v1 лимит является информационным и контрольным механизмом, а не hard-block. Если новая заявка выводит пользователя за лимит, заявка создаётся, пользователь получает предупреждение, а менеджер видит это в админке. Жёсткая блокировка заявок лимитом в этот пакет не входит. Это сохраняет логику исходного ТЗ, но без лишних SLA- и эскалационных надстроек. 

### 3.5. Ограничения whitelist

* максимум 5 адресов в статусах `pending` и `active` суммарно;
* адрес после отклонения нельзя повторно подать через UI/API;
* разрешённые сети: TRC-20, ERC-20, BEP-20.

Эти ограничения согласованы с текущим продуктовым описанием поддерживаемых сетей и с уже предложенной логикой whitelist.  

### 3.6. Уведомления

Поддерживаются только каналы `telegram` и `email`.
Пользователь может включать и выключать уведомления по типам событий:

* `order_created`
* `order_status_changed`
* `support_reply`
* `limit_warning`

Пользовательские настройки управляют только уведомлениями самому пользователю. Они не влияют на внутренние уведомления менеджерам.

## 4. Изменения данных

### 4.1. `ExchangeUserDB`

Добавить embedded-блок `notification_preferences`:

* `telegram_enabled: bool`
* `email_enabled: bool`
* `events: { order_created, order_status_changed, support_reply, limit_warning }`

Поле `sms_enabled` не добавлять. Текущая идея SMS-заглушки удаляется из этой редакции.  

### 4.2. `limit_quotas`

Новая коллекция:

* `user_id`
* `verification_level`
* `daily_limit`
* `daily_used`
* `daily_reset_at`
* `monthly_limit`
* `monthly_used`
* `monthly_reset_at`
* `updated_at`

Индекс: уникальный `user_id`. 

### 4.3. `limit_quota_history`

Новая коллекция только для админского аудита:

* `user_id`
* `changed_by`
* `field`
* `old_value`
* `new_value`
* `reason`
* `created_at`

Клиентский интерфейс эту историю не показывает. 

### 4.4. `whitelist_addresses`

Новая коллекция:

* `id`
* `user_id`
* `network`
* `address`
* `address_normalized`
* `label`
* `status` = `pending | active | rejected`
* `rejection_reason`
* `verified_by`
* `verified_at`
* `created_at`
* `updated_at`

Уникальный индекс: `user_id + network + address_normalized`. 

### 4.5. `OrderDB`

Добавить поля:

* `address_source: whitelist | manual`
* `whitelist_address_id: Optional[str]`
* `wallet_address`
* `wallet_network`

Если заявка создана на адрес из whitelist, `whitelist_address_id` обязателен. Это нужно для аудита и для доказуемой связи заявки с одобренным адресом. Исходное состояние без такой связи уже отмечено как проблема.

## 5. API

### 5.1. Клиентские эндпоинты

* `GET /api/profile/limits` — текущие дневной и месячный лимиты, использование и остаток;
* `GET /api/profile/notifications` — текущие настройки уведомлений;
* `PUT /api/profile/notifications` — обновление настроек уведомлений;
* `GET /api/profile/whitelist` — список адресов whitelist;
* `POST /api/profile/whitelist` — добавление нового адреса в статусе `pending`;
* `PUT /api/profile/whitelist/{id}` — изменение только `label`;
* `DELETE /api/profile/whitelist/{id}` — удаление адреса, если он не используется активной заявкой.

Клиентского `GET /api/profile/limits/history` в v1 нет. Это сознательно исключено. 

### 5.2. Админские маршруты

В `web/routers/users.py` добавить:

* страницу модерации whitelist;
* действия approve/reject для адреса;
* страницу редактирования лимитов пользователя;
* действие сохранения лимитов с записью в `limit_quota_history`.

Это соответствует текущей роли админ-панели, где уже сосредоточены пользовательские и модерационные сценарии.

## 6. Требования к интерфейсам

### 6.1. Dashboard

На `/dashboard/settings`:

* убрать моки и подключить реальные API;
* показать блок лимитов с дневным и месячным прогрессом;
* показать блок уведомлений с сохранением состояния;
* показать whitelist со статусами `pending / active / rejected`;
* для `rejected` показывать причину и CTA в поддержку;
* в публичных и dashboard-текстах использовать единое правило: сделки только на активные whitelist-адреса.  

### 6.2. Telegram-бот

* команда `/profile` показывает дневной и месячный лимит, использование и остаток;
* при создании заявки пользователь выбирает адрес из активного whitelist;
* ручной ввод адреса допускается только как шаг добавления нового адреса в whitelist;
* если адрес не активирован, бот не завершает создание заявки и предлагает отправить адрес на модерацию.

Это закрывает текущий разрыв, где в боте нет whitelist-проверки и нет продуктового отображения лимитов.  

### 6.3. Админка

* отдельная таблица pending-адресов;
* approve/reject с обязательной причиной при reject;
* отдельная форма редактирования лимитов;
* просмотр текущих лимитов и базовой истории изменений.

Без SLA-счётчиков, без эскалаций, без дополнительных приоритетных сортировок в v1. 

---

## 7. Нефункциональные требования

* все новые адреса проходят валидацию формата по сети на уровне фронта, API и бота;
* изменения лимитов пишутся в аудит;
* операции обновления использования лимитов выполняются атомарно;
* сброс дневного лимита — в 00:00 UTC, месячного — в 00:00 UTC первого числа месяца;
* производительность `GET /api/profile/limits` — не хуже p95 < 100 ms;
* пользовательские уведомления отправляются асинхронно и не должны блокировать основной flow создания заявки.  

---

## 8. Критерии приёмки

Пакет считается реализованным, если одновременно выполнены условия:

1. Пользователь в dashboard и в боте видит дневной и месячный лимиты с фактическим использованием.
2. Настройки уведомлений сохраняются и влияют только на пользовательские уведомления.
3. Пользователь может добавить whitelist-адрес, увидеть его статус и удалить его до активации.
4. Менеджер может одобрить или отклонить адрес через админку.
5. Заявка не может быть создана на адрес, который не находится в `active` whitelist.
6. В заявке сохраняется связь с whitelist-адресом или факт ручного ввода.
7. Изменение лимита через админку создаёт запись аудита.

Эти критерии закрывают все проблемы, явно перечисленные в анализе текущего состояния.
